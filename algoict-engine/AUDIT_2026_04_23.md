## 🔧 4 code fixes deployed LIVE today

### Fix #1: PWH/PDH forming-bar (04:39 ET)
**Problem**: `get_pdh_pdl / get_pwh_pwl` used `df.iloc[-1]` which grabbed the CURRENT FORMING week/day. Telegram reported PWH@27,138 but Topstep chart showed PWH=26,883 (−255pts discrepancy).

**Root cause**: tf_manager's weekly aggregation labels bars by Monday of ISO week. Current forming week was Apr 20 Monday; an overnight spike on Tue Apr 22 16:09 ET polluted that forming bar's high.

**Fix**: Added `as_of_ts` param to `build_key_levels / get_pdh_pdl / get_pwh_pwl`. When provided, filters out forming week/day before taking iloc[-1].

**Validation**: seeded levels post-fix matched chart exactly (PWH@26,883 ✅).

**Backward compat**: preserves legacy behavior when `as_of_ts=None` (backtester's per-window seeding unaffected).

### Fix #2: Phantom cleanup cooldown (12:18 ET)
**Problem**: After _poll_position_status detected "entry limit NEVER FILLED" and cancelled all 3 orders, the strategy immediately re-evaluated on the next bar. Same FVG/sweep/structure still valid → re-fired same signal. Loop: 5 fires in 27 min.

**Fix**: Added `_phantom_cooldown_until` attribute to SB + NY AM strategies + `record_phantom_cleanup(ts)` method. After cleanup, main.py calls `strat.record_phantom_cleanup(ts, cooldown_minutes=5)`. Strategy's evaluate() rejects `phantom_cooldown` while timestamp < cooldown end.

**Validation**: ZERO new fires since 12:18 ET (2h+ clean).

### Fix #3: TRADE OPENED alert gated on real fill (12:18 ET)
**Problem**: `_execute_signal` sent Telegram "Trade opened alert" the moment orders were submitted, regardless of fill status. User received 5 false "TRADE OPENED" alerts during phantom fires.

**Fix**: `_execute_signal` only sends alert if `entry_order.filled_price` is set to a real price. Otherwise logs "DEFERRED". `_on_broker_fill` sends the alert when entry fill is confirmed by broker.

**Validation**: No "TRADE OPENED" misleading alerts post-fix (bot had 0 fires post-fix so no positive confirmation, but code path is correct — tested via 1,477 unit tests).

### Fix #4: Single-position guard (13:26 ET)
**Problem**: `main.py._evaluate_strategies` checked `pending_signal_ts` (in-flight concurrency) and `risk.can_trade()` but NOT `state.open_positions`. If a trade was open, next bar could fire another signal and stack positions.

**Fix**: Added `if state.open_positions: continue` check in both NY AM + SB branches of _evaluate_strategies. Blocks new fires while any position (pending or filled) is already tracked.

**Validation**: Untested live (no fires post-VPIN-halt in NY PM). Unit tests pass (1,477/1,477).

## 🐛 Open issues identified during audit

### Minor bug: KZ CLOSED ny_pm never fires when hard close at same tick
main.py line 2273 `if state.hard_close_done: return` short-circuits before `_evaluate_strategies` runs. The KZ transition detection lives inside `_evaluate_strategies`, so on bar after hard close, transition from ny_pm → none is never detected.

**Fix options**:
- Move KZ transition detection BEFORE the hard_close check
- Or fire KZ CLOSED ny_pm as part of _flatten_all when reason=daily_hard_close

**Priority**: Low. Cosmetic — no trading impact. Tomorrow's NY PM same issue.

### Entry limit never-fill pattern
5 fires with entry limit above market → none filled. Root cause: ICT-canonical entry at FVG proximal edge is ±1 tick from FVG bottom/top. For short setups, that's "sell at FVG bottom" (ABOVE current price since FVG is below the price peak). Price needs to retrace to entry.

**This is BY DESIGN per ICT methodology** — "better to miss than chase". But when retrace doesn't happen, we waste broker API calls + phantom cleanup cycle.

**Fix options considered**:
1. Keep as-is (ICT canon) ← current behavior
2. Switch to market orders (user rejected earlier — changes strategy character)
3. Check price vs entry distance before placing orders (skip if > N pts away)
4. Reduce TTL on limit to 3 bars instead of 10 (cancel sooner)

**Decision**: Keep as-is. Phantom cooldown prevents re-fire spam. 1 failed attempt per setup is acceptable.

### Entry price calculation at PDH boundary
Fire #1 entry was 27,139.25 but bar high was 27,136.75 — price actually *didn't* sweep PDH 27,138 strictly (wick at 27,136.75 was below). But bot's sweep detector flagged PDH as SWEPT anyway. This suggests `check_sweep` uses `>=` or wick-above tolerance.

Worth verifying: was it a phantom sweep? Let me check detector logic in next audit cycle.
